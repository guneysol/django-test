30-second pitch

“BookNest is a community book-review platform built with Django. Members sign up, add books, write star-rated reviews, and curate a reading list. It has full CRUD, authentication and permissions, search/filter, pagination, a REST API, AJAX, and a third-party integration with the Open Library API for real cover art. Data lives in a relational database through Django’s ORM, the UI is server-rendered with a custom responsive design, and it’s deployable to Render with one blueprint.”

⭐ Where is the database?

It’s a SQLite database — a single self-contained file named db.sqlite3 sitting in the project’s root folder.

📁 /Users/guneyuzel/code/other/django-proj/db.sqlite3  (~216 KB)

Why SQLite?

SQLite is the database Django uses by default. It needs no separate server to install or run — the whole database is just that one file. That makes it perfect for development and demos.

Where is that configured?

In booknest/settings.py, the DATABASES setting:

DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.sqlite3",
        "NAME": BASE_DIR / "db.sqlite3",      # ← the file, in the project root
    }
}
Smart answer if they push on “is SQLite production-ready?”: “For a class project, yes — but the settings are environment-aware. If a DATABASE_URL environment variable is present (which it is on Render), the app automatically switches to PostgreSQL with zero code changes. So the same codebase runs SQLite locally and Postgres in production.”

The chain: Model → Migration → Table

catalog/models.py catalog/migrations/0001… db.sqlite3 ┌──────────────┐ makemigrations ┌──────────────┐ migrate ┌──────────────┐ │ class Book │ ───────────────▶ │ instructions │ ────────▶ │ catalog_book │ │ title = … │ │ to build the │ │ table with │ │ author = … │ │ table │ │ columns │ └──────────────┘ └──────────────┘ └──────────────┘
  1. You describe data as a model (Python class) in models.py.
  2. python manage.py makemigrations turns that into a migration file (the “blueprint” for the table).
  3. python manage.py migrate runs the migration, which creates/updates the real table in db.sqlite3.
  4. From then on, the ORM reads/writes rows in that table when you call things like Book.objects.all().

Q&A — Database

Where is the database located?

It’s a single SQLite file, db.sqlite3, in the project root. SQLite is serverless — the database is that file.

If they dig deeper: it’s configured in settings.py under DATABASES["default"], and switches to PostgreSQL automatically when a DATABASE_URL env var is present (production).

How does your Python code talk to the database? Do you write SQL?

Through Django’s ORM (Object-Relational Mapper) — I almost never write raw SQL. I write Python like Book.objects.filter(genre=g) and Django generates the SQL for me. This is safer (prevents SQL injection) and database-agnostic (works on SQLite and Postgres alike).

How did the tables get created?

Via migrations. I define models in models.py, run makemigrations to generate migration files, and migrate to apply them — which builds the actual tables. 21 migrations are currently applied.

How do you handle a many-to-many relationship in the DB?

With a join table. The reading list (users ↔ books) is a ManyToManyField. Django creates a hidden link table — catalog_book_shelved_by — where each row pairs one user id with one book id.

How do you keep the data consistent / prevent duplicates?

Database-level constraints, not just app code. A UniqueConstraint(book, author) guarantees one review per person per book; rating is validated 1–5; slugs are forced unique. There’s even a test that expects an IntegrityError to prove the constraint is real.

Isn’t querying slow / inefficient?

I optimised the read paths. I use select_related to avoid the N+1 query problem, and compute average rating & review count with a single annotated query (Avg/Count) reused for both display and sorting. Title and author columns are indexed for fast search.

Q&A — Models & relationships

What are your main models and how do they relate?

Four: Genre, Book, Review, and a Profile on top of Django’s User.

RelationshipType
User ↔ ProfileOne-to-One
Book → GenreForeignKey (many books, one genre)
Book → User (added_by)ForeignKey (the contributor)
Book ↔ User (reading list)Many-to-Many
Review → Book & Review → UserTwo ForeignKeys
Why did you make a separate Profile model instead of changing the User?

To extend the user without replacing Django’s battle-tested auth User. A one-to-one Profile holds extra fields (bio, avatar, location). A signal auto-creates the profile whenever a user is created, so it always exists.

What is a slug and why use it?

A URL-friendly version of the title, e.g. dune-frank-herbert. It makes clean, readable URLs instead of /books/7/. I generate it automatically and guarantee it’s unique.

What happens to reviews if a book is deleted?

They’re deleted too — the Review→Book link uses on_delete=CASCADE. But deleting a genre uses SET_NULL, so books survive and just lose their genre.

Q&A — Authentication & security

Are passwords stored in plain text?

No — never. Django hashes them with PBKDF2-SHA256 (1,200,000 iterations) plus a per-user salt. The DB stores something like pbkdf2_sha256$1200000$…, which can’t be reversed back into the password.

What’s the difference between authentication and authorization here?

Authentication = “who are you?” (signup/login/logout via Django sessions). Authorization = “what can you do?” — e.g. @login_required on pages that change data, and an ownership check so only a book’s contributor (or staff) can edit/delete it. The same rules are enforced in the REST API via a custom IsAuthorOrReadOnly permission.

How do you protect forms from CSRF attacks?

Django’s CSRF protection is on by default — every form includes a {% csrf_token %} and Django rejects POSTs without a valid token. My AJAX call sends the token in the X-CSRFToken header.

How does the site remember I’m logged in?

Session cookies. On login, Django creates a row in django_session and gives the browser a session-id cookie. Each request looks that id up to identify the user. In production, cookies are marked Secure + HttpOnly.

Q&A — Features & logic

How does search and filtering work?

It reads the URL query string (?q=dune&genre=fantasy&sort=rating) and builds one ORM query: a Q-object search across title/author/description, a genre filter, and a whitelisted sort. Then it’s paginated 6 per page.

What is AJAX and where do you use it?

AJAX = JavaScript talking to the server in the background so the page updates without a full reload. The “Add to reading list” button posts in the background and updates just the button + counter. The view detects the AJAX header and returns JSON; if JavaScript is off, the same button still works via a normal form submit (progressive enhancement).

How does pagination work?

Django’s Paginator slices the queryset into pages of 6, and the page links preserve the current search/filter/sort so you don’t lose your place.

Tell me about the third-party API integration.

I integrate the Open Library API. A management command (import_openlibrary) fetches real book data, and book covers are pulled live from Open Library’s Covers API by ISBN. The network code is isolated in services.py and is mocked in tests so the suite stays fast and offline-safe.

How does the Insights / charts dashboard work?

The server runs aggregation queries, the browser draws the charts. Each dataset (rating distribution, books per genre, average rating per genre, reviews-over-time, top books) is one ORM query using Count/Avg/TruncMonth — no Python counting loops. The view passes the numbers to the template as JSON (json_script), and Chart.js renders them client-side. So the charts are always live, never hard-coded.

Where does the demo data come from?

A custom management command: python manage.py seed_data. It creates genres, 14 books, demo users and 28 reviews. --flush resets it. It’s idempotent (safe to re-run).

Q&A — REST API

What is the REST API and how is it built?

A JSON interface to the same data, built with Django REST Framework at /api/. ViewSets + a router expose /api/books/, /api/reviews/, /api/genres/. Serializers convert models ↔ JSON (the API’s equivalent of templates).

Who can use the API? Is it secured?

Reads are public; writes require login (IsAuthenticatedOrReadOnly), and you can only modify your own records (object-level IsAuthorOrReadOnly). Ownership is stamped from the logged-in user, so a client can’t spoof it.

Can you show it?

Open /api/ in the browser for the clickable interface, or run:

curl "http://127.0.0.1:8000/api/books/?search=dune&ordering=-published_year"

Q&A — Deployment & testing

How would you deploy this?

To Render.com using the included render.yaml blueprint. It provisions a web service + a free PostgreSQL DB, runs build.sh (install → collectstatic → migrate), serves with Gunicorn, and serves static files via WhiteNoise. Security hardening (HTTPS redirect, HSTS, secure cookies) turns on automatically when DEBUG=0.

What’s the difference between dev and production settings?

The settings are environment-driven. Locally: SQLite, DEBUG=1, a dev secret key. In production: DEBUG=0, a real secret key, Postgres via DATABASE_URL, and all the security flags. Same code, different env vars. python manage.py check --deploy passes with zero issues.

How did you test it?

29 automated tests covering models, the unique-review constraint, every view, permission boundaries (a non-owner’s edit is rejected), search/filter/pagination, the AJAX endpoint, the REST API, signup/login/logout, and the Open Library integration (mocked). Run with python manage.py test. I also manually verified every flow in the browser.

Tricky / gotcha questions (with safe answers)

“SQLite isn’t a real database, is it?”

It absolutely is — it’s a fully relational SQL database, the most widely deployed database in the world (every phone uses it). For multi-user production traffic I’d use PostgreSQL, which my settings already support via DATABASE_URL. For a project demo, SQLite is the right, zero-setup choice.

“What happens if the Open Library API is down?”

The site is unaffected. Covers are loaded by the browser, and if one fails an onerror handler shows a book-emoji placeholder. The import command wraps failures in a clear error and the tests mock the network, so nothing depends on it being up.

“How is this MVC?”

Django calls it MVT: Model (data/ORM), View (the logic — equivalent to a controller), Template (the HTML). The framework itself plays the “controller” role of routing requests to views.

“Why two apps instead of one?”

Separation of concerns. accounts handles identity (profiles, signup); catalog handles the book domain. Each stays cohesive and could be reused independently — that’s the Django “app” philosophy.

“Where’s the business logic — in the templates?”

No. Templates only display data. Logic lives in views (request handling, permissions) and models (data rules, computed properties like average_rating), with external calls isolated in services.py. Templates stay “dumb.”

“If you had more time, what would you add?”

Good honest answer: review comments/likes, full-text search (Postgres), user following, email verification on signup, and caching for the book list. Shows you understand the next steps.
