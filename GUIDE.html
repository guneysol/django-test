1 · What the app is

BookNest is a website where a community of readers can keep a shared catalogue of books and talk about them.

Think of it as a tiny Goodreads. A logged-in member can:

📖 Add & manage books

Create a book with a title, author, genre, cover image, year and description — then edit or delete the ones they added.

⭐ Review & rate

Give any book a 1–5 star rating plus a written review. One review per person per book; posting again updates it.

🔖 Build a reading list

“Shelve” books they want to read. The button updates instantly without reloading the page (AJAX).

🔍 Search & browse

Search by title/author, filter by genre, and sort by rating, popularity or newest — with pagination.

👤 Profiles

Each member gets a public profile page listing their reviews and shelved books.

🔌 An API

The same data is available as a JSON REST API for programs, not just the browser.

The theme is just a vehicle. The point of the project is to demonstrate the Django skills your rubric asks for — models, views, templates, forms, auth, an API, and so on. “Books & reviews” is simply a domain rich enough to show all of them naturally.

2 · The big picture — how a request flows

Django uses the MVT pattern: Model (data) → View (logic) → Template (HTML). Here’s what happens when you click a book:

Browser ──GET /books/dune-frank-herbert/──▶ booknest/urls.py │ (matches the catalog routes) ▼ catalog/urls.py │ routes to the right view ▼ catalog/views.py → book_detail() │ asks the database via the ORM… ▼ catalog/models.py (Book, Review) ◀──▶ Database │ …gets the book + its reviews ▼ templates/catalog/book_detail.html (fills in the HTML) │ ▼ Browser ◀────────── finished HTML page ──────────────────────────────┘

Every page on the site follows that same path. The REST API follows an almost identical one, except the last step produces JSON instead of HTML (via a serializer rather than a template).

4 · The data model (the most important part)

Everything revolves around four tables. Here’s how they relate:

User ◀──1:1──▶ Profile (each account has one profile) │ │ added_by (who created it) ▼ Book ──ForeignKey──▶ Genre (each book has one genre; a genre has many books) │ ▲ │ └── shelved_by ◀── Many-to-Many ──▶ User (reading lists) │ │ 1 book has many… ▼ Review ── belongs to one Book AND one User ── UNIQUE(book, author): one review per person per book

Why these relationship types?

RelationshipTypeWhy
User ↔ ProfileOne-to-OneAdd fields (bio, avatar) to a user without replacing Django’s built-in User.
Book → GenreForeignKey (many-to-one)A book is in one genre; a genre groups many books.
Book → User (added_by)ForeignKeyTracks the owner so only they (or staff) can edit/delete it.
Book ↔ User (shelved_by)Many-to-ManyA reading list: many users shelve many books.
Review → Book & UserTwo ForeignKeysA review links one book to one author.
Data integrity is enforced by the database, not just code. The rule “one review per member per book” is a real UniqueConstraint, ratings are validated to 1–5, and slugs (the URL-friendly names like dune-frank-herbert) are guaranteed unique. A test deliberately triggers an IntegrityError to prove the constraint works.

A peek at the code

# catalog/models.py  (simplified)
class Review(models.Model):
    book   = models.ForeignKey(Book, on_delete=models.CASCADE, related_name="reviews")
    author = models.ForeignKey(User, on_delete=models.CASCADE, related_name="reviews")
    rating = models.PositiveSmallIntegerField(           # 1..5, validated
                 validators=[MinValueValidator(1), MaxValueValidator(5)])
    headline = models.CharField(max_length=120)
    body     = models.TextField()

    class Meta:
        constraints = [                                  # DB-level rule
            models.UniqueConstraint(fields=["book", "author"],
                                    name="unique_review_per_user_book")
        ]

5 · Feature walkthrough

CRUD — Create, Read, Update, Delete

Books and reviews can both be fully managed. The view checks ownership before any change:

# Only the contributor or a staff member may edit a book
if book.added_by != request.user and not request.user.is_staff:
    messages.error(request, "You don't have permission to edit this book.")
    return redirect(book)

Authentication & Authorization

Authentication = “who are you?” (signup / login / logout, powered by Django’s auth system). Authorization = “what are you allowed to do?” (the ownership check above, and @login_required on every page that changes data).

Search, filter, sort & pagination

The browse page reads the URL’s query string (?q=dune&genre=fantasy&sort=rating&page=2) and builds one efficient database query. Average rating and review count are calculated in that same query (an “annotation”), so sorting by rating doesn’t cause extra lookups.

books = Book.objects.select_related("genre").annotate(
    avg_rating=Avg("reviews__rating"),       # computed in SQL, reused for sort + display
    num_reviews=Count("reviews"),
)
paginator = Paginator(books, 6)              # 6 books per page

AJAX — the instant “Add to reading list” button

When you click shelve, JavaScript sends a background request and updates just the button — no page reload. If JavaScript is off, the same button still works via a normal form submit. That’s called progressive enhancement.

# The view detects an AJAX call and returns JSON instead of a redirect
if request.headers.get("x-requested-with") == "XMLHttpRequest":
    return JsonResponse({"shelved": shelved, "count": book.shelved_by.count(), ...})

Third-party API integration (Open Library)

A command can pull real book data off the internet from the free Open Library API:

python manage.py import_openlibrary "ursula k le guin" --limit 3 --dry-run

6 · The REST API

Built with Django REST Framework. Visit /api/ in the browser for a clickable interface. Reads are public; writing requires logging in, and you can only change your own records.

MethodEndpointDoes
GET/api/books/List books — supports ?search=, ?ordering=, ?page=
POST/api/books/Create a book (login required)
GET/PUT/DELETE/api/books/{slug}/Read / update / delete one book (owner only)
GET/POST/api/reviews/List / create reviews
GET/api/genres/List genres (read-only)

Try it from a terminal:

curl "http://127.0.0.1:8000/api/books/?search=dune&ordering=-published_year"

7 · How to run & test it

Run the site

source venv/bin/activate          # activate the virtual environment
python manage.py runserver        # start the dev server
# → open http://127.0.0.1:8000/

Run the automated tests

python manage.py test     # → Ran 29 tests … OK

python manage.py seed_data --flush
If you ever start fresh on another machine: pip install -r requirements.txt → python manage.py migrate → python manage.py seed_data → runserver.

9 · Mini Django glossary

TermPlain-English meaning
ModelA Python class that describes a database table (e.g. Book).
MigrationAn auto-generated file that tells the database how to build/change those tables.
ORM“Object-Relational Mapper” — lets you query the DB with Python (Book.objects.filter(...)) instead of raw SQL.
ViewA function that handles a request and decides what to send back.
TemplateAn HTML file with placeholders Django fills in with data.
FormValidates and cleans user input before it touches the database.
URLconfThe map from a web address to the view that handles it.
SerializerThe API’s version of a template — turns a model into JSON and back.
SlugA URL-safe name, e.g. the-hobbit-tolkien.
SignalA hook that runs code when something happens (we use one to auto-create a Profile).
AJAXJavaScript talking to the server in the background so the page updates without reloading.
